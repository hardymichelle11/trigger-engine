## Session 30: Phase 4.5A1 — Session-Aware Freshness Policy

### Trigger
After Phase 4.5A unblocked the universe, the next blocker surfaced: rejection storm reading `Quote data is stale · age ~900–1700s exceeds 900s for mode 'neutral'`. Live preview returned 0 candidates.

### Diagnosis
- The freshness field used (`updated`, falling back to `lastTrade.t`) was correct.
- Live probe of NVDA/AAPL/QQQ at 15:24 ET (regular session): all three reading age = 900–901 s. Polygon's snapshot endpoint is itself ~15 min delayed on the current plan tier; the 900 s neutral gate was rejecting nearly everything by exactly 1–3 seconds.
- Premarket / postmarket / closed sessions inherently produce older data (session-bounded) — a "live freshness" gate is meaningless.
- `sessionAwareUniverse` already wrote `metadata.universe.session`, but `marketDiscoveryScanner` only consumed `scannerMode`. Session was not propagated.

### What was built
- `src/engines/discovery/freshnessPolicy.js`: pure helper. `resolveFreshnessPolicy({ session, scannerMode })` returns either `{ maxStaleSec, source, justification }` or `{ disabled: true, … }` for closed. Frozen lookup table: regular conservative 1080 s / neutral 1500 s / aggressive 3600 s; premarket 21600 s; postmarket 21600 s; closed disabled. `isPriorSessionTimestamp(ageSec, session)` — true when regular-session age > 21600 s (catches yesterday's quotes during regular hours).
- `src/engines/discovery/marketDiscoveryScanner.js`: accepts `args.session` (preferred) or `args.regimeContext.session` (fallback). Stale check uses policy-resolved threshold; closed sessions skip the check entirely. Prior-session safeguard added before the mode-aware threshold.
- `src/components/discovery/LethalBoardPage.jsx`: one-line forward `session: bundle.metadata?.universe?.session` into `runMarketDiscoveryScan`. Metadata forwarding, not UI hardcoding. Verified by grep that no UI file contains any threshold literal.
- `scripts/test-discovery-phase4-5a1.js` — 72 assertions covering lookup table, mode ordering, mode-default back-compat, prior-session safeguard predicate, scanner integration across all 4 sessions × 3 modes, both `args.session` and `regimeContext.session` paths, page wire-through, source-code audit, purity audit, Phase 4 STALE-fixture regression.

### Decisions made
- **Backend + UI wire-through in the same phase (option B)** rather than splitting into 4.5A1 + 4.5A2. The forwarding line is metadata only, not threshold logic, so it's compatible with the "no UI hardcoding" rule.
- **Mode-default back-compat preserved.** Callers that don't pass session still get the legacy 300/900/3600 s thresholds — Phase 4 test fixtures continue to behave identically.
- **Prior-session safeguard at 6 hours.** Above all regular-mode thresholds, well below 24 h, so it catches yesterday's data without catching legitimate aggressive-mode (3600 s) data.

## Session 32: ThetaData v3 architectural decision

### Context
Phase 4.5C and Phase 4.5C+1 shipped earlier in the day targeting the **legacy local Theta Terminal v2** model (base `http://127.0.0.1:25510/v2`). When the operator went to install Theta Terminal locally and bind ports, a code snippet from current ThetaData docs surfaced a different scheme — `http://127.0.0.1:25503/v3/...` — that conflicted with the v2/25510 assumption baked into `thetaDataProvider.js`.

A docs-only investigation (no code changes) confirmed:

- **v2 endpoints return HTTP 410 GONE on a v3 Terminal.** They are not available side-by-side.
- ThetaData ships **two distinct access models**, not one:
  - **Local Theta Terminal v3** — Java app on operator's machine, exposes `127.0.0.1:25503/v3/...`. Terminal authenticates upstream at startup using a `creds.txt` file alongside the jar; **no credentials on the wire from the browser.**
  - **Cloud-direct REST / v3 Python library** — connects directly to ThetaData servers, requires email/password (or equivalent) in process. Not browser-safe; would require a backend proxy.
- **Subscription tiers gate the data, not the model.** All tiers require Terminal running. Real-time options snapshot quotes require **VALUE or higher**; FREE is EOD-only from 2023-06-01.

### Decision

**Target the local Theta Terminal v3 model.** Same architecture as Phase 4.5C — REST-only, browser → `127.0.0.1`, no secrets in the bundle — but on the v3 base URL and v3 path scheme. The cloud-direct model is deferred indefinitely; it would require a Node backend the dashboard does not currently have.

### Critical v2 → v3 deltas (for the upcoming refactor)

| | v2 (current adapter) | v3 (target) |
|---|---|---|
| Base | `http://127.0.0.1:25510/v2` | `http://127.0.0.1:25503/v3` |
| Health probe | `/v2/list/exchanges` | no documented dedicated probe; treat any non-410 from a list endpoint as alive (final path TBD during build) |
| Expirations | `/v2/list/expirations?root=SYM` | `/v3/option/list/expirations?symbol=SYM` |
| Snapshot quote | `/v2/snapshot/option/quote?root=SYM&exp=YYYYMMDD&strike=140000&right=C` | `/v3/option/snapshot/quote?symbol=SYM&expiration=YYYYMMDD&strike=140.00&right=put&format=json` |
| Strike scaling | provider-side `* 1000` (millicents) | dollars (no scaling) |
| Right normalization | `C` / `P` | `call` / `put` |
| Param naming | `root`, `exp` | `symbol`, `expiration` |
| Default response | CSV | CSV; pass `format=json` for browser use |

### Local setup requirements

- **Java 21+** (v3 Terminal will not run on lower)
- `ThetaTerminalv3.jar` (downloaded from operator's ThetaData account)
- `creds.txt` next to the jar — line 1: account email, line 2: password (file is local; never enters the app, the repo, or `VITE_*` env)
- `java -jar ThetaTerminalv3.jar` — Terminal authenticates upstream and binds local HTTP on `127.0.0.1:25503`

### Constraint reaffirmed

- **No real ThetaData credentials in `VITE_*` env variables.** `VITE_*` is only for non-secret browser-safe config: enable flag, base URL, timeout. Email/password live in `creds.txt` consumed by the Terminal, never by the app bundle.

- Terminal install / getting started: <https://docs.thetadata.us/Articles/Getting-Started/Getting-Started.html>

### Live integration trail (what we hit and how)

1. **Java 17 → Java 26 install.** v3 Terminal jar's class file version is 65 (Java 21+). Java 17's max is 61. `UnsupportedClassVersionError` until Java 21+ landed.
2. **`creds.txt` content fix.** First version had literal `line 1:` / `line 2:` prefixes copy-pasted from instructions. 3. **Terminal `config.toml` auto-generated** at first launch in project root. Added to `.gitignore` (machine-specific; would otherwise be tracked).
4. **CORS gap.** Terminal v3 emits **no `Access-Control-Allow-Origin` header** on responses. Direct browser fetch from `http://localhost:5173` to `http://127.0.0.1:25503` is blocked even though Terminal serves 200 OK. **Fix**: Vite dev-server proxy at `/theta` (same-origin from browser; Vite injects `Access-Control-Allow-Origin: http://localhost:5173` on the proxied response).
5. **v3 snapshot response shape mismatch.** Real v3 `/v3/option/snapshot/quote` ships a nested envelope:
   ```
   { response: [{ contract: {...}, data: [{ bid, ask, ... }] }] }
   ```
   Phase 4.5C+2 parser walked into `response[0]` and read `bid`/`ask` directly off the wrapper → undefined → `status: unavailable` even when valid quotes were one level deeper. **Fix**: detect `{contract, data: [...]}` shape in `parseThetaDataV3SnapshotPayload` and unwrap to `row.data[0]` before reading.
